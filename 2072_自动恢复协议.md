# 《2072》改写工程 自动恢复协议

> **重要：** 这份文件是每一次新会话进入本项目时 AI 必须读的第一份文件。
> **作者本人无须读此文件**——这是 AI 的工作流程文档。
> **作者只需在新会话开局说一句"继续 2072 改写"**，AI 即应自动按本协议恢复执行。

---

## 一、新会话开局的标准动作（30 秒）

无论多久之前的会话，AI 收到"继续 2072 改写"或类似指令后，立即按以下顺序执行：

**步骤 1：定位项目目录。**
项目目录在用户机器上：`C:\Users\kilro\OneDrive\Documents\Claude\Projects\2072`
当前 VM 路径请用：`/sessions/<本会话>/mnt/2072/`

**步骤 2：读四份"状态文件"（顺序固定）。**
```
1. 2072_进度看板.md          ← 当前完成到哪里、下一步是什么、有无阻塞
2. 2072_执行计划_v2.md       ← 整体阶段分解与任务清单
3. 2072_决策日志_v1.md       ← 已裁定决策清单（避免重复回炉）
4. MEMORY.md（自动加载）     ← 项目背景与高层状态
```

读完这四份文件，AI 应该能回答：
- 当前在第几阶段？
- 下一个任务是哪一章节？
- 有没有等待作者裁定的阻塞？
- 上次结束时记录了什么继续点？
- 哪些方向性问题已经裁定过、不需要再问？

**步骤 3：读"工具集"作为本次写作的工作内存。**
```
4. 2072_文风手册_v1.md       ← 写作规范（十规、红线、把握度梯度）
5. 2072_代号对照表_v1.md     ← 代号→中文意象名映射
6. 2072_史鉴对照表_v1.md     ← 历史平行素材库
7. 2072_新目录草案_方向B_v1.md  ← 全书结构图
```

如果本次任务专注于某一具体章节，可只读相关部分而非全文，节省上下文。

**步骤 4：向作者确认起点。**
开局简短报告："上次进度到 X，下一步建议执行 Y。是否继续？"——给作者一个确认机会。如果作者要求改方向，按改后方向走。

## 二、单次会话内的工作流程

**写一章的标准动作：**

1. 在执行计划任务清单中找到本章任务，把状态从"待定"改为"进行中"，记录开始时间。
2. 读 v143 该节原文（用 python-docx）。
3. 检索史鉴对照表中对应主题的可用历史平行。
4. 检索代号对照表替换该节出现的代号。
5. 按文风手册十规起草，写入 `v144_chapters/卷X章YY_标题.md`。
6. 自查质量门禁五项（见下文）。
7. 在进度看板更新该章状态为"已完成"，记录字数、本次会话编号、完成时间。
8. 如果本会话还有上下文余量，回到步骤 1 推进下一章。
9. 会话结束前，**最后一件事是更新进度看板**——记录"本会话完成 N 章，下次从 X 继续"。

**质量门禁五项（每章完成前必查）：**

- [ ] 文风手册十规：作者声音 1—2 次、加粗 3—5 处、把握度标注、句式节奏、无硬交叉引用。
- [ ] 代号对照表：本章所有代号已替换为中文意象名，首次出现处附括注一次。
- [ ] 史鉴对照表：本章至少调用一条真实历史平行（地理章节可放宽至每章一条而非每节）。
- [ ] 红线五条：无虚构具名个体、推论与历史分清、未为可读性放弃精确、未过度文学化、未夹带个人偏好。
- [ ] 字数：在估算 ±20% 范围内（见执行计划任务清单）。

任一不通过则不得标"已完成"，标"返工"或"待审"。

## 三、阻塞与中断处理

**遇到需作者裁定的事情：**

**第一步：先查决策日志。** 同类问题可能已被裁定，避免重复回炉。
**第二步：若日志中无对应裁定**，不要自己拍板。在进度看板的"阻塞清单"里写一行新条目，状态置"等待作者"，并停下当前章节工作。
**第三步：作者裁定后**，立即把新裁定追加到决策日志（编号 D-XXX 递增），同时清除阻塞条目。

格式：
```
- [日期] [章节号] 问题：XXX；建议方案：XXX；建议默认：XXX
```

**遇到上下文用尽：**

- 如果当前章节已完成：标记状态、更新看板、结束。
- 如果当前章节未完成：把已写部分保存为 `卷X章YY_标题_unfinished.md`，在看板记录"未完成，已写约 Z 字，剩余 W 字"，下次会话从断点继续。

**遇到工具失败（Write 报错、bash 报错等）：**
切换到另一种工具（Write 失败用 bash heredoc，bash 失败用 Write）。如果两种都失败，记录到看板"工具阻塞"并停止。

```
├── 2072_进度看板.md                       # 进度（每次会话末更新）
├── 2072_自动恢复协议.md                   # 本文件
├── v144_chapters/                       # 改写章节工作目录（成品）
│   ├── 卷0序一_01_2027临界年.md

└── v144_volumes/                        # 卷级合并稿（按需生成）
    ├── 卷0_v1.md

```

**章节文件命名规则：** `卷{X}章{YY}_{标题简称}.md` 或 `卷{X}序{字}_{编号}_{标题}.md`
- X：卷号 0—6
- YY：章号两位数（章 01—章 99，便于排序）
- 序：仅卷 0 使用
- 标题简称不超过 6 字，避免文件名过长

**示范文件保留**（已完成的三份示范作为基线，不改）：
- `2072_文风改写_1.4.1示范.md` → 对应卷 2 章二·2.1
- `2072_文风改写_3.14.6示范.md` → 对应卷 1 章八·8.2.6
- `2072_文风改写_6.7.5示范.md` → 对应卷 5 章八·8.5

正式改写时把示范内容拷入正式章节文件，并按需修订衔接段。

## 五、进度看板更新规范

**每次会话至少更新两次：**
- 开局：在"本会话计划"区写下本次拟做的章节列表。
- 收尾：在"已完成章节"区添加新条目；在"下一步建议"更新指针。

**字段必须包含：**
- 章节号
- 标题
- 字数
- 本会话编号
- 完成日期
- 质量门禁是否全过（Y/N）

---

## 六、Memory 更新触发条件

只在以下两种情况下更新 MEMORY.md：

1. **完成一个完整阶段**（Phase 1—10 中的任何一个），简要记录"Phase X 已完成，时间 YYYY-MM-DD"。
2. **关键裁定变更**（作者推翻或修改之前的裁定），即时记录避免下次会话遗忘。

**不要为单章完成更新 MEMORY.md**——那是进度看板的工作。MEMORY.md 是用来跨会话保留高层状态的，不是日记。

---

## 七、本协议的迭代

如果在执行过程中发现本协议的某条规则不好用，**可以在进度看板的"协议迭代建议"区记录**，但本协议本体的修订需要作者确认后再改。
