# 词条摘要/概述写作规范（草案 v2）

**状态**：D-051③ 裁定材料（2026-07-15 起草 v1；同日按 Sean 意见修订 v2——新增可验证性、压缩/抽象区分、删减优先、反例库、summary 来源裁量收敛五项）。适用于全部词条的 `summary`（一句话摘要）与 `overview`（一段概述）字段——渐进披露的第一、二层，可读性的实际承重层。原型 9 条为参照样本。

## 0　总则

1. 摘要与概述是**压缩，不是改写**（D-045）：从原文提取并浓缩，不得引入原文没有的论断、数字、因果关系或价值判断。第三层（原文）零改动，由构建按节号提取。
2. **压缩 ≠ 抽象**（v2 新增）：允许删减细节，不允许提升抽象层级。
   - 原文链：A→B→C→D。
   - 允许（压缩）："A导致D"——中间环节省略，因果链条本身未变形。
   - 禁止（抽象）："A体现了制度惯性"——这是对原文的重新归类/上位概括，跨了一层，即使"读起来是对的"也不可以。
   - 判断标准：压缩后的句子若不能逐词映射回原文的具体因果链，只是"提炼出一个更高级的说法"，即为抽象，判定不合格。
3. **删减优先于改写**（v2 新增）：能删就删，不能删才改写句式。
   - 原文："四层制度是在工业时代逐步形成的。"
   - 优先："四层制度在工业时代逐步形成。"（删减虚词，句式不变）
   - 避免："四层制度源于工业时代。"（"逐步形成"→"源于"看似同义，实际弱化了"逐步"这一时间性论断，是改写而非删减）
   - 只有在原句因删减后语法不通或指代不明时，才允许最小幅度的改写，且改写后仍须通过第 4 条的可追溯性检验。

## 1　语域

1. 分析语域的简明形态：陈述句、判断句为主，与源文档同一语域，只是密度更低。
2. 禁止：文学化修辞、悬念句式（"……会发生什么？"）、营销语气（"深刻揭示""令人惊讶"）、叙事化开场（"想象一下"）。
3. 禁止自指：不以"本节""本词条""这一部分"开头——词条会在悬停预览、索引卡片等脱离上下文的场合被读到。
4. 机制归因优先于描述性标签（与正文同一纪律）：摘要若给结论，须是原文可追溯的机制命题，不是概念简写。

## 2　可验证性（Verifiability，v2 新增）

1. summary 中的每一个机制判断，都必须能定位到原文中的一句或连续几句——不能定位即视为该 summary 不合格，无论读起来是否"像是对的"。
2. frontmatter 增加可选字段 `source`，标注该 summary/overview 论断在原文中的位置（段落序号或首尾锚句），便于日后复核：
   ```yaml
   summary: "..."
   source: "§4.9.1 第3段"
   ```
   长词条（overview 综合多段）可给区间，如 `"§4.9.1 第2–4段"`。
3. 本条的目的是把"作者应该是这个意思"这类模型推断挡在门外——可验证性不是锦上添花，是防止未来更强模型用更流畅的语言掩盖不可追溯的推断。

## 3　summary（一句话摘要）

1. 长度：推荐 ≤45 字；超过 60 字视为不合格（v2 放宽——语言有天然长度，不强行卡在45字，但60字是硬上限）。单句，可用一个破折号或冒号分层。
2. 内容 = 该词条**作者最终确立、且全文反复支撑的机制命题**（v2 修订，替换 v1 的"最核心的机制论断"——原表述留给不同模型太大裁量空间，同一词条不同模型会选中不同"核心"）。判定优先级：
   - 若原文存在斜体结论段（coda），默认以其机制命题作为 summary 的来源，不再另行判断"什么最核心"；
   - 若无 coda，选择原文中被其他节引用/依赖次数最多的那条命题（可参考 related/backlink），而非撰写者主观认为最重要的一条。
   - 差："本词条介绍A1体系的社会保障结构。"（主题描述）
   - 好："A1的社会保障不是重新设计的，而是在工业时代遗留框架上叠加改造成的四层结构。"（机制论断，且可追溯到原文 coda 或高频引用句）
3. 概念词条的 summary = 定义压缩（"是什么"），章节词条的 summary = 论点压缩（"为什么/如何"）。
4. 索引词条的 summary = 本节覆盖范围一句话（自动生成场合亦遵守）。

## 4　overview（一段概述）

1. 长度 80—160 字，单段，不分点。
2. 结构建议：核心机制一句 → 关键展开/阶段/数字一至两句 → 边界或张力一句。
3. 数字、区间、年份**照抄原文**，不四舍五入、不改写单位；原文给区间的不得只取一端。
4. 原文的斜体结论段（coda）是 overview 的优先素材——那已是作者自己的压缩。
5. 术语首现用全称（如"基本支持积分（BSC）"），概念链接由构建自动生成，正文中不手写链接。
6. A1/A2 对照内容遵守 A1 先于 A2 的文档约定。
7. 不做跨词条综合：overview 只压缩本词条对应原文，即使写作者知道其他节有相关内容（跨词条关系交给 related 字段与正文链接）。

## 5　反例库（Anti-pattern，v2 新增）

以下句式一律禁止，出现即判定该 summary/overview 不合格，不论内容是否正确——这些是模型最容易产出、最难人工逐条排查的失败模式：

| 禁止句式 | 问题 |
|---|---|
| "本文分析……"/"本节讨论……" | 违反 1.3 自指禁令 |
| "深刻揭示……"/"令人惊讶的是……" | 营销/文学语气，违反 1.2 |
| "可以看出……"/"由此可见……" | 引入了"推导过程"的叙述腔，原文若无此推导措辞即为改写 |
| "作者认为……"/"作者指出……" | 把机制命题包装成"作者观点"，弱化了命题本身的断言力度，且原文未必如此措辞 |
| "……表明……"/"这意味着……" | 常用于把压缩偷换成抽象（见0.2）——"A表明B"往往是"A→B"因果链被重新归类为解释关系 |
| "值得注意的是……" | 无信息量的引导句，纯粹增字数 |
| "想象一下……"/悬念句式 | 违反 1.2 |
| 任何四舍五入或改写单位的数字表述 | 违反 4.3 |

## 6　质量控制流程

1. frontmatter 增加 `reviewed` 字段（初值 no）：AI 生成 → 作者复核语域与论断保真 → 改 yes。check_entries 可统计复核进度。
2. 原文哈希漂移告警（check_entries [D] 项）触发后：先复核 summary/overview 是否仍成立，再 `--update-lock`。
3. 抽查基线：每批切分随机抽 10% 由作者盲读 summary，若能从 summary 猜错原文立场（引入了原文没有的倾向），该批整体返工。
4. **可追溯性抽查**（v2 新增）：抽查时同时核对 `source` 字段——随机抽取的 summary，作者需能在标注位置找到对应原文句，找不到即判定为该批返工信号，不只是单条问题（说明生成流程系统性地产生了不可追溯的推断）。

## 7　各词条类型的差异要点

| 类型 | summary 重心 | overview 重心 | 特殊约束 |
|---|---|---|---|
| 概念 | 定义（是什么） | 定义+首要机制角色+边界 | 正文=词典定义原文+机制展开映射，不重复推导 |
| 章节 | 核心机制论断 | 机制+关键数字+张力 | 正文=构建提取，零改写 |
| 事件 | 该示例演练了什么机制 | 时间线+设置变量+可替换性 | 接口声明字段齐全才算完成 |
| 索引 | 覆盖范围 | 各子词条一句话串联 | 构建自动生成，规范约束生成模板 |
